# Extractor Registry Generation

**`registry_gen.go` in this directory is MANUALLY MAINTAINED** — it is not output from an automated code generator despite its header comment.

## Do Not Hand-Edit (Outside of Registry Maintenance)

The file contains blank imports of all language extractor packages. While it is manually edited during development, **any significant restructuring or reorganization should preserve its structure**.

The file header claims "Code generated by go:generate; DO NOT EDIT" for clarity that it is a registry — not a typical source file — but edits happen directly in the file during feature development.

## How to Add a New Extractor

When creating a new language extractor in `internal/extractors/<lang>/`:

1. **Create the extractor package** at `internal/extractors/<lang>/` with an `init()` function that calls `extractor.Register()`.

2. **Add a blank import** to `registry_gen.go`:
   ```go
   _ "github.com/cajasmota/grafel/internal/extractors/<lang>"
   ```

3. **Keep imports in alphabetical order** (enforced by CI via `gofmt -w`).
   - After you edit, run `gofmt -w internal/extractors/registry_gen.go` or just `gofmt -w .` to auto-sort.

4. **Update the classifier** in `internal/classifier/` to recognize file extensions for the new language.

5. **Add resolver patterns** in `internal/engine/rules/<lang>/` if needed (see sister file `internal/extractors/custom_registry.go` for the pattern of framework-specific extensions).

## Why This File Exists

The registry is a **bootstrap problem**: every extractor registers itself via `init()` when imported, but `main` packages cannot import them all without creating a pseudo-arbitrary dependency. Blank imports in `registry_gen.go` (imported by `registry.go`, which is a normal package) ensure all extractors are linked and their `init()` functions run.

Unlike `custom_registry.go` (which wires framework-specific customizations), `registry_gen.go` is the **language baseline registry**. Each blank import ensures that language's extractor is available on startup.

## Maintenance

- **No CLI tool**: There is no `make reggen` or equivalent. Edits are made directly.
- **Alphabetical sorting**: CI runs `gofmt` on all Go files (see `.github/workflows/test.yml`), which automatically keeps imports alphabetically sorted. Any PR that breaks alphabetical order will fail the gofmt CI step.
- **When to regenerate**: You never regenerate this file outright — you add one import per new language and re-sort via `gofmt`.

## Testing

The registry is tested in `registry_test.go`:

- Verifies that all registered languages can be looked up
- Spot-checks extraction on representative fixtures
- Validates that `init()` functions fired correctly (via `Register()` calls)

Run tests with:

```bash
go test ./internal/extractors/... -v
```
